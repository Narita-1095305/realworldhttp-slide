第9章 HTTP/2時代の新しいユースケース

Webの進化とブラウザを超えた通信の世界

検索エンジンやソーシャルメディアが理解するプロトコルを中心に、HTTP/2時代の新技術を解説します。

レスポンシブデザイン (1/2)

CSSピクセルとは？

デバイスの物理的な解像度（物理解像度）とは別に、CSSがレイアウト計算に使う論理的な解像度のこと。

例: iPhone 4は物理640x960pxですが、論理的には320x480pxとして扱われます。

デバイスピクセルレシオ

物理解像度と論理解像度の比率です。高解像度ディスプレイほど値が大きくなります。

例: Nexus 6Pはレシオが3.5。つまり、1つのCSSピクセルが縦横3.5個の物理ピクセルで描画されます。

JavaScriptでの確認

ブラウザの開発者ツールで以下のコードを実行すると、現在のウィンドウサイズとピクセルレシオを確認できます。

// 実行例 (Nexus 6Pの場合)
> window.innerWidth
412
> window.innerHeight
732
> window.devicePixelRatio
3.5

レスポンシブデザイン (2/2)

CSSメディアクエリ

画面幅に応じてスタイルを切り替えます。

/* デスクトップ向け */
@media screen and (min-width: 480px) {
  .text { color: blue; }
}

/* モバイル向け */
@media screen and (max-width: 479px) {
  .text { color: red; }
}

viewport設定

論理ピクセル幅をデバイスの幅に合わせるためのHTMLのmetaタグです。

<meta name="viewport" content="width=device-width, initial-scale=1.0">

複数画像の出し分け

デバイスピクセルレシオに応じて最適な解像度の画像を表示します。

<img src="photo.png" srcset="photo@2x.png 2x, photo@3x.png 3x">

WebP

Googleが開発。JPEG/PNG比で約2/3のファイルサイズ。非可逆・可逆・透過・アニメーション全てに対応する多機能フォーマット。

AVIF

AV1動画コーデックベース。WebPよりさらに圧縮率が高く、HDRにも対応。現在主要ブラウザでサポートされている。

HEIF

H.265(HEVC)動画コーデックベース。iOSの標準フォーマット。AVIFと同等の圧縮率を誇るが、Webブラウザでの対応は限定的。

セマンティックウェブ

定義

Webの提唱者ティム・バーナーズ・リーが構想した、コンピュータがWebページの意味（セマンティクス）を理解し、処理できるようにする仕組み。

現在の実現方法

1. ボキャブラリーの利用

用途ごとに決められた語彙（例: schema.org）をHTMLに埋め込み、機械に意味を伝える。

2. 機械学習の活用

ChatGPTのようなAIが文脈から意味を推定し、構造化されていない情報も理解する。

マイクロデータ

HTMLタグに属性を追加することで、セマンティックな情報を埋め込むための仕様です。

  • itemscope: 意味の対象範囲を定義
  • itemtype: schema.orgなどのボキャブラリーで型を定義
  • itemprop: 個々のプロパティ（名前、説明など）を定義

schema.org の記述例

<div itemscope itemtype="http://schema.org/Person">
  <span itemprop="name">ジョージ・ブッシュ</span>：
  <span itemprop="disambiguatingDescription">第41代アメリカ大統領</span>
  <div itemprop="children" itemscope itemtype="http://schema.org/Person">
    <span itemprop="name">ジョージ・W・ブッシュ</span>：
    <span itemprop="disambiguatingDescription">第43代アメリカ大統領</span>
  </div>
</div>

JSON-LD

Googleが推奨する構造化データ形式。HTMLの見た目に影響を与えず、<script>タグでJSONデータを埋め込めます。

これにより、検索結果での表示がリッチになり、クリックされる前に多くの情報をユーザーに提供できます。

scriptタグでの埋め込み例

<script type="application/ld+json">
{
  "@context": "http://schema.org",
  "@type": "Organization",
  "url": "http://www.your-company-site.com",
  "contactPoint": {
    "@type": "ContactPoint",
    "telephone": "+1-401-555-1212",
    "contactType": "customer service"
  }
}
</script>

RSS

RDF (Resource Description Framework) の成功例の一つ。ウェブサイトの更新履歴をまとめたXMLベースのフォーマットです。

ユーザーはRSSリーダーを使うことで、複数のサイトの更新情報を効率的にチェックできます。

RSS 1.0 の実例コード

<rdf:RDF xmlns:rdf="http://www.w3.org/1999/02/22-rdf-syntax-ns#"
         xmlns:dc="http://purl.org/dc/elements/1.1/"
         xmlns="http://purl.org/rss/1.0/">
  <channel rdf:about="http://blog.shibu.jp/">
    <title>Shibu's Diary</title>
    <link>http://blog.shibu.jp/</link>
    <items>
      <rdf:Seq>
        <rdf:li rdf:resource="http://blog.shibu.jp/article/178099358.html"/>
      </rdf:Seq>
    </items>
  </channel>
  <item rdf:about="http://blog.shibu.jp/article/178099358.html">
    <title>passive な #kaneの話</title>
    <description>これはPySpa アドベントカレンダー2016の...</description>
    <dc:creator>@shibukawa</dc:creator>
    <dc:date>2016-12-20T01:20:30+09:00</dc:date>
  </item>
</rdf:RDF>

vCard

連絡先情報（名刺情報）を交換するための標準フォーマットです (RFC 6350)。MIMEタイプは text/vcard です。

vCardサンプル

BEGIN:VCARD
VERSION:3.0
N:渋川;よしき;;;
TEL:(+81) 3-3356-5227
ADR:;;四谷坂町12-22;新宿区;東京都;160-0002;日本
END:VCARD

実行結果

この形式のファイル（.vcf）をデバイスで開くと、連絡先アプリが起動し、情報のインポートを促します。

iCalendar

スケジュール情報（予定）を交換するための標準フォーマットです (RFC 5545)。MIMEタイプは text/calendar です。

iCalendarサンプル

BEGIN:VCALENDAR
VERSION:2.0
BEGIN:VEVENT
DTSTART:20190713T000000Z
DTEND:20190713T010000Z
SUMMARY:GoConference '19 summer in Fukuoka
DESCRIPTION:Go and Enjoy Fukuoka!
LOCATION:Fukuoka Growth Next
URL:https://fukuoka.gocon.jp/ja/
END:VEVENT
END:VCALENDAR

実行結果

この形式のファイル（.ics）をデバイスで開くと、カレンダーアプリが起動し、予定の追加を確認されます。

オープングラフプロトコル (OGP)

Meta (旧Facebook) が開発。SNSなどでWebページのリンクが共有された際に、リッチな表示（タイトル、画像、説明文など）を実現するための仕組みです。

OGPを元にしたTwitterカードもあり、両方を記述することで主要なSNSに対応できます。

実践的な実装例 (Qiitaの場合)

<html xmlns:og="http://ogp.me/ns#">
<head>
  <meta charset="UTF-8" />
  <!-- Twitter Card -->
  <meta name="twitter:card" content="summary" />
  <meta name="twitter:site" content="@Qiita" />
  <meta name="twitter:creator" content="@shibu_jp" />

  <!-- OGP (Facebook, Twitter共通) -->
  <meta property="og:title" content="最速MVCフレームワーク Mithril.jsの速度の秘密 - Qiita" />
  <meta property="og:type" content="article" />
  <meta property="og:url" content="http://qiita.com/shibukawa/items/..." />
  <meta property="og:image" content="http://cdn.qiita.com/assets/qiita-fb.png" />
  <meta property="og:description" content="Mithril 0.2が本日リリースされました..." />
  <meta property="og:site_name" content="Qiita" />

  <!-- Facebook用 -->
  <meta property="fb:admins" content="554524838" />
</head>
</html>

QRコード: 物理世界との連携

QRコードとは？

デンソーウェーブが開発した2次元バーコード。大容量のデータを格納でき、誤り訂正機能も強力です。

単なる文字列だけでなく、「スキーム」を解釈して特定のアプリを起動できます。

多様なユースケース

  • https://...: Webサイトを開く
  • mailto:...: メーラーを起動
  • tel:...: 電話を発信
  • WIFI:...: Wi-Fiに接続
  • BEGIN:VCARD...: 連絡先を登録
  • BEGIN:VEVENT...: カレンダーに予定を登録

セキュリティ上の注意

URLを隠蔽できるためフィッシングに悪用される危険性も。安易に不明なQRコードを読み取らないよう注意が必要です。

DeepLink: Webとアプリの連携

DeepLinkとは？

特定のURL（リンク）をトリガーに、Webサイトではなくネイティブアプリの特定の画面を直接開く仕組みの総称です。

  • iOS: Universal Link
  • Android: intentフィルター (App Links)

これにより、Webとアプリをシームレスに遷移させ、ユーザー体験を向上させます。

X (旧Twitter) の例

以下の形式のURLにアクセスすると、Xのアプリがインストールされていれば投稿画面が直接開きます。

https://twitter.com/intent/tweet?
  url=<共有したいURL>
  &text=<本文>
  &hashtags=<ハッシュタグ>

アプリがなければ、通常のWebサイトの投稿ページが表示されるフォールバックも用意されています。

HLS (HTTPライブストリーミング)

Appleが2009年に提唱した動画配信技術 (RFC 8216)。

特徴:

  • リアルタイム中継に対応
  • 回線状況に合わせて解像度を自動選択
  • 字幕や多言語音声の切り替えが可能

HTMLでの利用例

.m3u8 というプレイリストファイルを指定します。

<video controls autoplay>
  <source src="http://example.com/video.m3u8"/>
  <source src="/fallback.ogv" type='video/ogg'>
</video>

HLS: マスタープレイリスト

解像度ごとや字幕・音声ごとのプレイリスト(.m3u8)をまとめた、マスターとなるプレイリストです。

#EXTM3U
#EXT-X-MEDIA:TYPE=SUBTITLES,GROUP-ID="subs",NAME="English",URI="subtitles/eng/prog_index.m3u8"
#EXT-X-STREAM-INF:BANDWIDTH=1193108,CODECS="avc1.77.30,mp4a.40.2",RESOLUTION=640x266
0640/0640.m3u8
#EXT-X-STREAM-INF:BANDWIDTH=6807315,CODECS="avc1.580028,mp4a.40.2",RESOLUTION=1920x800
1920/1920.m3u8

HLS: 動画セグメントリスト

マスタープレイリストから参照される、特定の解像度の動画を構成するセグメントファイル（.ts）の一覧です。

#EXTM3U
#EXT-X-TARGETDURATION:13
#EXT-X-PLAYLIST-TYPE:VOD
#EXTINF:12.012,
0640_00001.ts
#EXTINF:12.012,
0640_00002.ts
...

HLS: 字幕ファイル (WebVTT)

W3Cのウェブ標準でもあるWebVTT形式の字幕ファイルです。タイムスタンプと表示するテキストで構成されます。

MPEG-DASH (1/2)

MPEGコンソーシアムによる動画配信の国際標準 (ISO/IEC 23001-6)。YouTubeなど多くのサービスで採用されています。

マニフェストファイル (MPD, Media Presentation Description) というXMLで配信内容を記述します。

MPDファイルの階層構造

  • Period: チャプターや広告など、時間的な区切り。
  • AdaptationSet: 同一コンテンツの代替セット（例: 映像、音声、字幕）。
  • Representation: 具体的なメディアストリーム（例: 高解像度映像、低解像度映像、英語音声）。
  • Segment: 動画や音声の断片ファイル。

MPEG-DASH (2/2)

MPDファイル詳細コード例

映像と音声のAdaptationSetが分かれ、それぞれに帯域ごとのRepresentationが含まれる構造を確認できます。

<?xml version="1.0" encoding="utf-8"?>
<MPD xmlns="urn:mpeg:dash:schema:mpd:2011" profiles="..." type="static" ...>
  <Period>
    <!-- 音声トラック -->
    <AdaptationSet mimeType="audio/webm" ...>
      <Representation id="171" codecs="vorbis" bandwidth="129553">
        <BaseURL>audio_128k.webm</BaseURL>
        <SegmentBase indexRange="...-..."/>
      </Representation>
      <Representation id="172" codecs="vorbis" bandwidth="188041">
        <BaseURL>audio_192k.webm</BaseURL>
        <SegmentBase indexRange="...-..."/>
      </Representation>
    </AdaptationSet>
    
    <!-- 映像トラック -->
    <AdaptationSet mimeType="video/webm" ...>
      <Representation id="245" codecs="vp9" width="854" height="480" bandwidth="1431499">
        <BaseURL>video_480p.webm</BaseURL>
        <SegmentBase indexRange="...-..."/>
      </Representation>
      <Representation id="247" codecs="vp9" width="1280" height="720" bandwidth="3915200">
        <BaseURL>video_720p.webm</BaseURL>
        <SegmentBase indexRange="...-..."/>
      </Representation>
    </AdaptationSet>
  </Period>
</MPD>

まとめ

レスポンシブデザイン: 複数解像度のスクリーンに対応する技術。
セマンティックウェブ: 構造化データで検索結果をリッチにする。
OGP: SNSでの表示を豊かにするメタ情報。
QRコード: URLだけでなく多様な情報を格納・連携。
動画ストリーミング: HTTP基盤で大容量動画を効率的に配信。
アプリ連携: DeepLinkでWebとアプリをシームレスに繋ぐ。

結論

これらの技術は、もはや人間だけがWebを見るのではなく、機械が情報を理解し、処理することで、最終的に人間のユーザー体験を向上させるために進化しています。
